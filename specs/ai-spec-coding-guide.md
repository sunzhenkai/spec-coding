# AI Spec Coding 规约指南

## 概述

Spec Coding（规格驱动编程）是一种通过编写详细的规格说明（Specification）来指导代码生成的编程方法。当与 AI 助手配合使用时，清晰、完整的规格说明能够显著提高代码生成的质量和准确性。

本规约旨在为 AI 辅助编程提供一套标准化的规格编写指南，帮助开发者更好地与 AI 协作。

---

## 1. Spec 编写规范

### 1.1 规格文档结构

一个完整的规格说明应包含以下部分：

#### 必需部分
- **标题与概述**：简要描述功能/模块的目的
- **输入/输出定义**：明确说明输入参数和预期输出
- **功能需求**：详细描述核心功能和行为
- **约束条件**：性能要求、资源限制、兼容性要求等

#### 可选部分
- **依赖关系**：外部库、API 或其他模块依赖
- **错误处理**：异常情况和错误处理策略
- **示例用例**：典型的使用场景和示例代码
- **测试要求**：需要覆盖的测试场景

### 1.2 编写原则

#### ✅ DO（推荐做法）
- **使用清晰、准确的语言**：避免歧义，使用专业术语
- **提供具体的数据类型和范围**：如 `age: number (0-150)` 而不是 `age: number`
- **使用结构化格式**：Markdown、JSON Schema、TypeScript 接口等
- **包含边界条件**：说明极端情况的处理方式
- **提供示例**：用具体例子说明预期行为

#### ❌ DON'T（避免做法）
- 使用模糊的描述，如"大约"、"可能"、"尽量"
- 遗漏错误处理场景
- 假设 AI 能理解隐含的上下文
- 提供矛盾的或不完整的需求

### 1.3 规格模板

```markdown
## [功能名称]

### 概述
[一句话描述功能目的]

### 输入
- `参数名`: 类型 - 描述
  - 约束: [可选/必需，范围，格式等]

### 输出
- 返回类型: 类型
- 描述: [输出内容的说明]

### 功能需求
1. [需求1]
2. [需求2]
3. [需求3]

### 约束条件
- 性能: [时间/空间复杂度要求]
- 兼容性: [支持的版本/环境]
- 安全性: [安全相关要求]

### 错误处理
- 场景1: [错误情况] -> [处理方式]
- 场景2: [错误情况] -> [处理方式]

### 示例
\`\`\`
输入: ...
输出: ...
\`\`\`

### 测试场景
- [ ] 正常情况测试
- [ ] 边界条件测试
- [ ] 错误情况测试
```

## 2. AI 提示技巧

### 2.1 提示词结构

有效的 AI 提示应遵循以下结构：

```
1. 上下文设定 (Context)
2. 角色定义 (Role)
3. 任务描述 (Task)
4. 规格说明 (Specification)
5. 约束条件 (Constraints)
6. 示例/参考 (Examples)
7. 输出格式 (Output Format)
```

### 2.2 提示技巧

#### 分步骤提供信息
不要一次性提供所有信息，而是：
1. 先描述整体架构和目标
2. 再提供详细的规格说明
3. 最后提供具体的实现要求

#### 使用代码块和类型定义
```
请实现以下接口:

\`\`\`typescript
interface UserService {
  // 根据ID获取用户信息
  getUserById(id: string): Promise<User | null>;
  
  // 创建新用户
  createUser(data: CreateUserDTO): Promise<User>;
}
\`\`\`

要求：
- 使用 async/await
- 包含参数验证
- 处理数据库连接错误
```

#### 提供上下文信息
```
项目背景:
- 这是一个电商系统的用户模块
- 使用 TypeScript + Node.js
- 数据库: PostgreSQL
- ORM: TypeORM
- 需要支持事务处理
```

### 2.3 常见提示模式

#### 模式1：从规格到实现
```
我需要实现一个[功能名称]，以下是详细规格：

[插入完整规格]

请根据以上规格：
1. 设计数据结构/接口
2. 实现核心逻辑
3. 添加错误处理
4. 编写单元测试
```

#### 模式2：迭代优化
```
这是当前的实现代码：
[代码]

存在以下问题：
1. [问题1]
2. [问题2]

请优化代码以解决这些问题，同时：
- 保持原有功能
- 遵循[编码规范]
- 提高代码可读性
```

#### 模式3：代码审查
```
请审查以下代码：
[代码]

审查重点：
- 是否符合规格要求
- 代码质量
- 潜在的bug
- 性能问题
- 安全隐患

请提供改进建议和优化后的代码。
```

## 3. 代码生成最佳实践

### 3.1 规格的粒度控制

#### 功能级规格
适用于独立的功能模块：
```markdown
## 用户登录功能

### 功能描述
验证用户凭据并返回认证令牌

### 输入
- email: string (有效的邮箱格式)
- password: string (8-64字符)

### 输出
- 成功: { token: string, user: UserDTO }
- 失败: Error

### 验证规则
1. 邮箱格式验证
2. 密码长度验证
3. 用户存在性检查
4. 密码匹配验证

### 安全要求
- 密码使用 bcrypt 加密
- 限制登录尝试次数 (最多5次/15分钟)
- 生成 JWT token (有效期24小时)
```

#### 模块级规格
适用于复杂系统的整体架构：
```markdown
## 用户管理模块

### 模块组成
1. UserService - 业务逻辑层
2. UserRepository - 数据访问层
3. UserController - API 控制器
4. UserValidator - 输入验证器

### 模块间依赖
UserController -> UserService -> UserRepository
                    ↓
                UserValidator

### 公共接口
- POST /api/users - 创建用户
- GET /api/users/:id - 获取用户
- PUT /api/users/:id - 更新用户
- DELETE /api/users/:id - 删除用户

### 数据模型
[详细的数据模型定义]
```

### 3.2 渐进式规格编写

不要试图一次性编写完整的规格，而是：

1. **MVP 规格**：最小可行产品的核心功能
2. **迭代增强**：逐步添加新功能和优化
3. **反馈修正**：根据实现反馈调整规格

```
版本 1.0: 基础 CRUD 操作
版本 1.1: 添加输入验证
版本 1.2: 添加权限控制
版本 2.0: 支持批量操作
```

### 3.3 规格与代码的同步

- **版本控制**：规格文档应纳入版本控制
- **变更日志**：记录规格变更历史
- **代码注释**：在代码中引用规格版本
- **自动化验证**：使用测试确保实现符合规格

## 4. 示例和测试

### 4.1 完整示例：任务队列

#### 规格说明
```markdown
## TaskQueue - 任务队列

### 概述
实现一个异步任务队列，支持任务优先级和并发控制

### 接口定义
\`\`\`typescript
interface Task<T> {
  id: string;
  priority: number; // 1-10, 数字越大优先级越高
  execute: () => Promise<T>;
}

class TaskQueue {
  constructor(concurrency: number);
  add<T>(task: Task<T>): Promise<T>;
  pause(): void;
  resume(): void;
  clear(): void;
  getStats(): QueueStats;
}
\`\`\`

### 功能需求
1. 支持并发执行，可配置最大并发数
2. 根据优先级调度任务
3. 支持暂停和恢复队列
4. 提供队列状态统计

### 约束条件
- 最大并发数: 1-20
- 队列容量: 最多10000个任务
- 任务超时: 默认30秒

### 错误处理
- 队列已满: 抛出 QueueFullError
- 任务超时: 抛出 TaskTimeoutError
- 任务失败: 不影响其他任务，记录错误

### 测试场景
\`\`\`javascript
// 1. 基本功能测试
const queue = new TaskQueue(3);
const result = await queue.add({
  id: 'task1',
  priority: 5,
  execute: async () => 'done'
});
assert(result === 'done');

// 2. 优先级测试
// 3. 并发控制测试
// 4. 暂停/恢复测试
// 5. 错误处理测试
\`\`\`
```

#### AI 提示示例
```
请实现一个任务队列系统，规格如下：

[插入上述规格说明]

技术栈：
- TypeScript
- 不使用外部库

要求：
1. 完整实现所有接口方法
2. 包含详细的代码注释
3. 提供单元测试（使用 Jest）
4. 处理所有边界情况

请分步骤实现：
第一步：定义接口和数据结构
第二步：实现核心队列逻辑
第三步：实现并发控制
第四步：添加错误处理
第五步：编写测试用例
```

### 4.2 测试驱动规格

在规格中明确定义测试用例：

```markdown
## 测试用例清单

### 单元测试
- [ ] `add()`: 正常添加任务
- [ ] `add()`: 队列已满时抛出错误
- [ ] `add()`: 无效参数时抛出错误
- [ ] `pause()`: 暂停后不执行新任务
- [ ] `resume()`: 恢复后继续执行
- [ ] `clear()`: 清空队列
- [ ] `getStats()`: 返回正确的统计信息

### 集成测试
- [ ] 并发限制正确生效
- [ ] 优先级调度正确工作
- [ ] 任务失败不影响其他任务
- [ ] 长时间运行稳定性

### 性能测试
- [ ] 1000个任务的吞吐量
- [ ] 内存使用在合理范围
```

## 5. 工作流程建议

### 5.1 推荐工作流程

```
1. 需求分析
   ↓
2. 编写初始规格 (Draft)
   ↓
3. 与 AI 讨论规格 (Review)
   ↓
4. 完善规格 (Refine)
   ↓
5. AI 生成代码实现
   ↓
6. 代码审查与测试
   ↓
7. 反馈修正规格 (如果需要)
   ↓
8. 文档更新
```

### 5.2 与 AI 协作的技巧

#### 首次对话
```
我正在开发一个[项目描述]。

技术栈：
- [语言/框架]
- [数据库]
- [其他依赖]

当前任务：
[具体任务描述]

请先确认你理解了需求，然后我会提供详细的规格说明。
```

#### 提供规格后
```
以下是完整的规格说明：

[规格内容]

请：
1. 先指出任何不清晰或矛盾的地方
2. 确认理解所有需求
3. 然后再开始实现
```

#### 迭代改进
```
这是当前实现：

[代码]

测试结果：
[测试输出/错误信息]

请：
1. 分析问题原因
2. 提供修复方案
3. 更新实现代码
```

### 5.3 规格文档管理

- **命名规范**: `spec-[功能名称]-v[版本号].md`
- **存放位置**: `/specs` 目录
- **变更记录**: 在文档顶部维护变更日志
- **关联代码**: 在代码注释中引用规格文档

## 6. 常见问题与解决方案

### Q1: 规格太详细，AI 生成的代码不够灵活？
**解决方案**: 规格应关注"做什么"（What）而非"怎么做"（How），给 AI 留出实现细节的空间。

### Q2: AI 生成的代码不符合项目规范？
**解决方案**: 在规格中包含编码规范说明，或提供示例代码作为参考。

### Q3: 规格变更频繁，如何管理？
**解决方案**: 使用版本控制，维护变更日志，使用增量式更新而非重写。

### Q4: 如何确保 AI 理解复杂的业务逻辑？
**解决方案**: 
- 使用图表或流程图辅助说明
- 提供具体的业务场景示例
- 分步骤解释复杂逻辑

### Q5: 生成的代码缺乏错误处理？
**解决方案**: 在规格中明确列出所有可能的错误场景和处理策略。
